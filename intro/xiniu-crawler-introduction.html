烯牛政策爬虫平台

项目介绍

本项目是"烯牛政策图谱"的采集与结构化基座：把分散在中央、国务院、各部委以及 31 省市自治区门户上的产业政策，自动、可追溯地汇集起来，原稿一字不落地留存，再用 AI 拆解为标准 36 字段 JSON，最终服务于 LP/GP、银行、券商、企业战投与研究机构的"检索 + 解读 + 统计 + 对比 + 推送"。

1. 项目要解决的问题

痛点

  • 政策来源分散在 100+ 网站，版面与字段各异
  • 政策正文、附件、元信息不统一，难检索难比对
  • 人工运营无法保证覆盖与时效
  • 合规与原稿可追溯性是金融客户的硬性要求

交付物

  • 稳定可订阅的政策数据流
  • 100% 可回放的原始页与附件
  • 36 字段标准 JSON（元数据 / 实体内容 / 标签）
  • 可审计：每个字段都能追到原稿与抓取批次

2. 总体架构：双平面设计

系统由"代码生成平面"与"执行平面"组成，两者通过同一个 Git 仓库串联——代码合并即上线。这种结构让"为新网站写适配器"这件繁琐的事可以由 AI 编程代理自动完成，而真正的抓取主路径仍是确定性代码，AI 不进入每一次请求。

图 1 · 双平面架构：AI 在离线写适配器，在线主路径仍是确定性代码

设计原则

  • 控制面自研，数据面复用成熟组件——抓取/解析/存储用经过验证的库，调度与契约自己掌握。
  • AI 只放在高不确定决策点——URL 排序、页面分类、反爬识别、字段抽取；规则优先，AI 兜底。
  • 抓取层级递进：feed/sitemap → 静态 HTML → 接口拦截（JSON/GraphQL）→ SSR/DOM → headless 渲染（按需）。命中信号才升级。
  • 原始与解析分离：所有原始字节 + HTTP 元数据落 OSS，可任意回放；解析层独立演进，不影响原稿。

3. 一条政策的生命周期

图 2 · 一条政策的生命周期：主链路 5 步，原稿与结构化数据分别归档
  1. 提交种子：人工或定时任务给出网站入口 URL。
  2. 礼貌下载：以 host 为最小限速单元，遵守 robots、Retry-After，遇 429/503 走"Retry-After 优先 + 指数退避 + 抖动 + cooldown"。
  3. 原稿落盘：原始字节直接写 OSS，附件优先按文件名展示。
  4. 解析：调用该 host 对应 adapter，输出列表/详情结构（标题、正文、发布日期、附件）。
  5. 去重：以 content_sha256 严格去重；simhash 仅作相似聚类信号，不自动合并。
  6. AI 抽取：将正文 + 网页元数据交给 LLM，输出 36 字段标准 JSON 并校验 schema。
  7. 入库：元数据进 PolarDB，原稿留 OSS，可随时回放与审计。

5. 业务规格摘要：政策数据是什么样的

第一阶段采集范围

  • 中央行政机关：国务院政策文件库 + 要闻
  • 国务院部委：覆盖 8 家目标（发改委 / 工信部 / 财政部 / 人民银行 / 金融监管总局 / 证监会 / 国资委 / 科技部），目前已落地 5 家适配器：ndrc / most / csrc / nfra / sasac
  • 地方机关：31 省市自治区"信息公开 + 要闻 + 政策解读"约 90 个 URL
  • 引用：国家法律法规数据库（仅跳转，不打标签）
  • 后期补齐：金融交易所、行业协会

每条政策的三个组成部分

  1. 元数据表格：网页自带的属性表
  2. 正文：政策主体文本
  3. 附件：PDF 等原始字节（本期不做 PDF→文本）

36 字段 JSON 的三段结构

  • 元数据字段（15 个）：标题、发文字号、发布机关、行政区划、政策工具分类等
  • 实体内容字段（9 个）：金额、期限、对象、条件等
  • 标签字段（12 个）：行业、主题、地域映射等

6. 合规与底线（不可妥协）

不做什么

  • 不绕过验证码 / 登录 / 付费墙 / 技术 challenge
  • 不使用专为绕过保护设计的工具或库
  • 不对 robots 明示拒绝的路径继续抓取（5xx 视为完全禁止）
  • 默认不抽身份证号、手机号、邮箱等敏感字段

怎么做

  • 识别异常 → 降速 / 暂停 / 人工审核
  • 每个采集任务必须填写：用途、责任人、范围
  • 从设计起预留 删除 / TTL 链路
  • warm-up 分级：新 host 按 10%/30%/60%/100% 阶梯放量，反爬命中即降到 L0+cooldown

7. 仓库结构与工程方法

项目采用 "单仓 + 文档先行 + 轻量 DDD + 规格驱动开发（SDD）" 模式，规格、计划、代码、评估证据全部在同一仓库内闭环，方便审计与回放。

xiniu-crawler/ ├── docs/ 控制面（架构、规格、计划、任务、评估） │ ├── prod-spec/ 11 份长期规格（业务 + infra + codegen） │ ├── exec-plan/active/ 当前执行计划（MVP / codegen-bootstrap / WebUI 等） │ ├── task/ active/completed/archive 工作流目录 │ ├── prd/ research/ 产品原稿与工程视角研究底稿 │ └── eval-test/ 评估与回放证据 ├── domains// 业务代码（按 bounded context） │ └── gov_policy/ 按部委拆分子模块：ndrc / most / csrc / nfra / sasac， │ 每个子模块自带 *_adapter.py + *_seed.yaml ├── infra/ 跨业务复用的技术能力（不含业务规则） │ ├── 采集运行时 crawl / http / robots / frontier / storage / render │ └── codegen 平台 agent / harness / sandbox / codegen / │ adapter_contract / adapter_registry / source_probe ├── webui/ FastAPI + React + Ant Design ProComponents ├── scripts/ 仓库级 CLI 入口 └── runtime/ 本地运行时（SQLite / 抓取缓存，gitignore）

工作流：每一次改动都走"规格 → 计划 → 任务 → 代码 → 证据"

  • Planner 锁定上下文与作用域，更新 spec，输出已批准的计划
  • Generator 实现单个原子任务，更新触达的文档与测试
  • Evaluator 独立验证，记录 green/red 证据，关闭任务状态
  • Cleaner 例行整洁、归档噪声，把非紧急债务推后

分支命名 agent/<type>-YYYYMMDD-<topic>，任务 ID T-YYYYMMDD-NNN，一支分支 → 一个活跃任务 → 一个 PR。
